Privacy Policy

Last updated: July 17, 2026

This public pilot is designed to use only the information needed to respond to inquiries you explicitly send and to operate artifact verification. Auxtho does not intentionally configure site analytics, advertising trackers, or advertising cookies. These statements describe the reviewed implementation and are not evidence that every hosting or security control is continuously effective. Hosting and security providers may process technical request data as part of delivery and abuse protection.

Current Auxtho verification QR URLs place the report ID, record binding checksum, and optional export event ID after the URL fragment marker (#). Browsers do not send that fragment to the site-serving, CDN, or origin infrastructure. Direct identifier query parameters on verify.html were retired on July 17, 2026. The page immediately removes them from the visible URL. Only an old query-form 16-character binding is recognized, solely to show a non-identifying local tombstone; its values are not copied into controls and the page starts no readiness or comparison API request. Other retired query identifiers are ignored after scrubbing. Previously issued API lookup URLs may still carry identifiers to the API in their path or query before the API redirects the browser to a fragment-based site URL. Opening or prefilling a current fragment-form page does not itself start an artifact comparison request or intentionally create an application verification audit record; delivery and security infrastructure may separately log the page and readiness requests.

When you explicitly select Run Verification or submit the manual verification form, accepted inputs are sent to Auxtho for comparison. The submitted report ID, record binding checksum, and optional export event ID are used for that request but are not copied into a verification application audit record when one is created.

If you optionally select a PDF for a file-bytes check, the file contents remain in your browser. The browser computes a SHA-256 digest and sends only that digest for comparison; the PDF itself is not uploaded, and the file-bytes digest is not written to the verification audit record.

If a comparison request reaches the instrumented application path and its audit write succeeds, the verification API creates a minimized record containing the outcome, reason code, time, expiry time, and a one-way SHA-256 fingerprint derived from the submitted identifiers. Client-side validation, API validation, rate limiting, network failure, or service unavailability may result in no application audit record. A click or submit action is not proof that an application audit record exists. When created, that record does not contain the submitted report ID, record binding checksum, export event ID, selected file digest, file contents, email address, or IP address.

Website, CDN, hosting, API, and security providers may separately process request URLs, IP addresses, user agents, request timing, routing data, and security signals in infrastructure logs. Those logs are not verification application audit records, and their fields and retention depend on the provider and deployment. Where the current pilot verifier creates an application audit record, it is intended to include an expiry timestamp 30 days after creation. Deletion occurs only when the deployment's Firestore TTL policy is configured and active, and must be verified operationally. This is a pilot data-minimization policy, not a regulatory-retention guarantee. Do not submit personal, customer-confidential, or secret identifiers; contact Auxtho for a deletion request or retention question.

The registered contracting-entity name, jurisdiction of formation, legal data-controller identity, and controller notice address are unresolved in the repository. Contract tests, release manifests, and deployment readbacks do not resolve that legal gap or prove continuous operating effectiveness. Do not use this site to submit regulated, customer-confidential, or personal data, and do not rely on this notice for regulated onboarding, until those fields are identified in an approved signed agreement and privacy notice.
